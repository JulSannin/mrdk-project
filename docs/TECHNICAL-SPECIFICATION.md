## 1. Общие сведения

### 1.1. Назначение документа
Настоящее техническое задание определяет назначение, состав, функциональные и нефункциональные требования, архитектуру, структуру данных, программные интерфейсы и требования к развёртыванию информационной системы МРДК.

### 1.2. Назначение системы
Система предназначена для:
- информирования жителей округа о деятельности Дома культуры (события, клубы и кружки, планы работы, документы, памятки, противодействие коррупции, контакты);
- публикации фото- и видеоматериалов прошедших и планируемых мероприятий;
- приёма обращений граждан через форму обратной связи;
- самостоятельного ведения контента сотрудниками учреждения без участия разработчика через административную панель.

### 1.3. Область применения
Публичный информационный сайт государственного (муниципального) учреждения культуры. Аудитория — жители Мариинского округа и посетители учреждения, в том числе люди с нарушениями зрения (предусмотрена версия для слабовидящих).

### 1.4. Термины и сокращения

| Термин | Расшифровка |

| МРДК | Условное обозначение системы / Муниципальный районный Дом культуры |

## 2. Цели и задачи создания системы

### 2.1. Цели
1. Предоставить учреждению современный, доступный и быстрый публичный сайт.
2. Обеспечить самостоятельное управление контентом силами сотрудников.
3. Соответствовать требованиям к сайтам государственных учреждений: доступность для слабовидящих, согласие на обработку персональных данных (152-ФЗ), раздел противодействия коррупции.

### 2.2. Задачи
- Реализовать публичную часть с разделами: события, клубы, план работы, документы, памятки, противодействие коррупции, контакты.
- Реализовать защищённую административную панель с CRUD-операциями над контентом.
- Обеспечить загрузку и хранение медиафайлов (изображения, видео) и документов.
- Интегрировать карту (2ГИС), веб-аналитику (Яндекс.Метрика), отправку обращений на e-mail.
- Обеспечить безопасное, автоматизируемое развёртывание и резервное копирование.

## 3. Характеристика объекта автоматизации

Дом культуры регулярно проводит мероприятия, ведёт кружки и клубные формирования, публикует нормативные документы и планы работы. Ранее контент вёлся на стороннем хостинге со старой панелью управления (rdk-mbuk.ru/cpanel). При создании системы выполнен перенос исторических данных: **593 события** с фотогалереями (~1300 изображений) и видеозаписями.

## 4. Роли пользователей

| Роль | Доступ | Возможности |
|---|---|---|
| **Посетитель (аноним)** | Публичная часть, без аутентификации | Просмотр всех публичных разделов, скачивание документов, отправка обращения, включение версии для слабовидящих |

Ролевая модель — двухуровневая. В БД роль хранится в поле `users.role` (по умолчанию `admin`). Регистрация пользователей через интерфейс не предусмотрена; учётная запись администратора создаётся автоматически при первом запуске бэкенда из защищённого секрета.

## 5. Функциональные требования

### 5.1. Публичная часть

#### 5.1.1. Главная страница (`/`)
- Блок «Последние события»: адаптивная сетка карточек (число карточек подстраивается под количество колонок: 4–12), ссылка «Все события →».
- Блок карточек внешних ресурсов (Культура.РФ, Минкультуры, Госуслуги, Гранты России, опрос и др.).
- Блок видео.

#### 5.1.2. События (`/events`)
- Постраничный вывод карточек событий (по 24 на страницу), пагинация в URL (`?page=N`).
- Фильтр по году проведения (`?year=YYYY`); список доступных годов со счётчиком подгружается отдельным запросом.
- Состояние фильтра и страницы сохраняется в URL (ссылка шарится, работают кнопки «назад/вперёд» браузера).
- Скелетоны при загрузке, приоритетная загрузка изображений первого ряда (оптимизация LCP), плавное появление.

#### 5.1.3. Карточка события (`/events/:id`)
- Заголовок, дата проведения, описание.
- Галерея изображений (главное + дополнительные).
- Блок «Видео» — встроенный плеер (`<video controls>`) с поддержкой потоковой отдачи (HTTP Range).

#### 5.1.4. Клубы и секции (`/clubs`)
- Список клубных формирований: наименование и руководитель. Сортировка по названию.

#### 5.1.5. План работы (`/workplan`)
- Список планов, сгруппированных/упорядоченных по году ↓ и месяцу ↓ (свежие сверху).
- Скачивание файла плана (PDF/DOC/DOCX) с сохранением оригинального имени файла.

#### 5.1.6. Документы (`/documents`)
- Список документов учреждения, скачивание файлов с корректным оригинальным именем (в т.ч. кириллица).

#### 5.1.7. Памятки (`/reminders`)
- Список памяток в виде изображений (заголовок + картинка).

#### 5.1.8. Противодействие коррупции (`/anticorruption`)
- Перечень нормативных правовых актов (внешние ссылки на pravo.gov.ru) и локальных документов учреждения (PDF из статики фронтенда).

#### 5.1.9. Контакты (`/contacts`)
- Реквизиты учреждения (наименование, директор, телефон, адреса, ИНН/КПП/ОГРН, e-mail), перечень филиалов.
- Интерактивная карта 2ГИС с маркерами учреждения и филиалов; при отсутствии ключа API — текстовая ссылка-фолбэк на 2ГИС.
- **Форма обратной связи**: имя, e-mail, телефон, тема, сообщение, обязательный чекбокс согласия на обработку ПДн. Клиентская и серверная валидация; отправка письма администратору.

#### 5.1.10. Служебные страницы
- `/login` — страница входа в админку.
- `*` (404) — страница «не найдено».
- Сквозной баннер согласия на обработку cookie и ПДн (152-ФЗ), запоминается в `localStorage`.

#### 5.1.11. Версия для слабовидящих (BVI)
Реализуется через панель BVI поверх публичных страниц (на админку не распространяется). Настройки сохраняются в `localStorage`. Параметры:
- **Размер шрифта**: обычный / крупный / очень крупный.
- **Цветовая схема**: белая / чёрная / синяя / бежевая / зелёная.
- **Межстрочный интервал**: обычный / средний / большой.
- **Межбуквенный интервал**: обычный / средний / большой.
- **Изображения**: показывать / оттенки серого / скрыть.
- Возможность скрыть саму панель.

Стилизация управляется через `data-*` атрибуты на корневом элементе `<html>`, что изолирует компоненты от логики режима.

#### 5.1.12. Общие детали интерфейса
- Скелетоны-заглушки при загрузке и плавное появление (fade-in) — на всех списках (события, памятки, документы, план, клубы), главной и карточке события; скелетон показывается мгновенно, анимация — только на контенте.
- Заголовок вкладки браузера (`<title>`) меняется по разделу, для события — по его названию.

### 5.2. Административная панель (`/admin`)

Реализована на базе **react-admin** (русская локализация). Доступ — только после входа; при отсутствии сессии — редирект на `/login`. Загружается отдельным код-чанком (посетители публичной части его не скачивают).

Ресурсы (для каждого — список, создание, редактирование, удаление):

| Ресурс | Поля | Файлы |
|---|---|---|
| **События** | Заголовок, описание, дата события | Главное изображение (необяз. — без него на сайте заглушка), доп. изображения, видео |
| **План работы** | Заголовок, год, месяц | Файл документа (обяз.) |
| **Документы** | Заголовок | Файл документа (обяз.) |
| **Памятки** | Заголовок | Изображение |
| **Клубы** | Название, руководитель | — |

Особенности админки:
- Управление галереей события: добавление пакета изображений/видео отдельными запросами, удаление отдельных файлов. При сбое догрузки медиа после создания события администратору выводится явное сообщение.
- Сортировка по столбцам поддерживается для событий; остальные ресурсы — фиксированный порядок.
- Ввод дат в российском формате (компонент `RuDateInput`).
- Скользящая сессия: активный администратор не разлогинивается во время работы (см. 8.1).

## 6. Архитектура и технологический стек

### 6.1. Общая архитектура
Трёхзвенная архитектура, разворачиваемая как набор Docker-контейнеров за обратным прокси nginx:

```
[Браузер] ──HTTPS──> [nginx] ──┬── статика SPA (dist)
                                ├── /uploads  (отдача файлов)
                                ├── /api  → проксирование → [backend:3001] ──> [PostgreSQL]
                                └── /health → backend
                                                    │
                                          [SMTP] <──┘ (обратная связь)
```

### 6.2. Монорепозиторий
Весь проект — один git-репозиторий, разворачивается клонированием на сервер и `docker compose up`.

```
work/
├── .github/workflows/    # CI: backend.yml, frontend.yml (только в корне)
├── mrdk-back/            # REST API
├── mrdk-front/           # SPA: публичный сайт + админка
├── nginx/nginx.conf      # прод-роутинг, CSP, проксирование
├── secrets/              # docker secret: admin_password (вне git)
├── docs/                 # документация (этот файл, OPERATIONS.md)
├── docker-compose.yml    # postgres + backend + nginx + certbot
├── deploy.sh             # деплой: git pull + build + up
├── backup.sh / restore.sh# резервное копирование и восстановление
└── .env                  # переменные окружения compose (вне git)
```

### 6.3. Стек технологий

**Backend (`mrdk-back/`)**
- Node.js 20, TypeScript (ESM), Express 5.
- PostgreSQL 16 через драйвер `pg` (пул соединений).
- Аутентификация: `jsonwebtoken` (JWT), `bcryptjs` (хеширование паролей).
- Загрузка файлов: `multer` (disk storage) + `file-type` (проверка реального типа по сигнатуре).
- Безопасность: `helmet` (в т.ч. CSP), `cors` (allowlist), `express-rate-limit`.
- Валидация: `express-validator`.
- Почта: `nodemailer` (SMTP).
- Логирование: `winston` + `morgan`.
- Запуск: dev — `tsx watch`; prod — `tsc` → `node dist/server.js`.
- Тесты: `vitest`. Линт: `eslint` + `typescript-eslint`.

**Frontend (`mrdk-front/`)**
- React 18, TypeScript, Vite 8.
- Маршрутизация: `react-router-dom` 6.
- Серверное состояние: TanStack Query (`@tanstack/react-query`).
- Админка: `react-admin` 5 + MUI, `ra-data-simple-rest`, `ra-language-russian`, `@mui/x-date-pickers`.
- HTTP-клиент: `axios` (с `withCredentials`).
- Пагинация: `react-paginate`. Даты: `dayjs`.
- Карта: 2ГИС (динамическая подгрузка). Аналитика: Яндекс.Метрика.
- Архитектура: **Feature-Sliced Design** — слои `app/`, `pages/`, `widgets/`, `entities/`, `shared/`.
- Тесты: `vitest`. Линт: `eslint`.

**Инфраструктура**
- Docker, docker-compose: `postgres:16-alpine`, backend (Node 20 alpine, non-root), nginx (`nginx:alpine`), `certbot/certbot`.
- Тома: `postgres_data`, `uploads`, `certbot_certs`, `certbot_www`.
- Docker secret: `admin_password`.
- CI: GitHub Actions (lint + typecheck + test, раздельно для back/front по path-фильтрам).

## 7. Структура данных

СУБД — PostgreSQL. Миграции — сырые SQL-файлы в `mrdk-back/migrations/`, применяются на старте бэкенда по порядку имени и фиксируются в таблице `schema_migrations` (повторно не применяются). Механизма наполнения данными (сидов) нет — контент вводится через админку или импортируется.

### 7.1. Таблицы

**`users`** — учётные записи (фактически только администратор)
| Поле | Тип | Примечание |
|---|---|---|
| id | SERIAL PK | |
| name | VARCHAR(100) | |
| login | VARCHAR(100) UNIQUE NOT NULL | |
| password_hash | VARCHAR(255) NOT NULL | bcrypt |
| role | VARCHAR(50) NOT NULL DEFAULT 'admin' | |
| created_at | TIMESTAMP DEFAULT NOW() | |

**`events`** — события
| Поле | Тип |
|---|---|
| id | SERIAL PK |
| title | VARCHAR(255) NOT NULL |
| description | TEXT |
| image_path | VARCHAR(500) — главное изображение |
| event_date | DATE |
| created_at | TIMESTAMP DEFAULT NOW() |
| updated_at | TIMESTAMP NULL |

Индекс: `idx_events_created_at (created_at DESC)`.

**`event_images`** — дополнительные изображения галереи
- id, event_id (FK → events, ON DELETE CASCADE), image_path NOT NULL, created_at.

**`event_videos`** — видеозаписи события
- id, event_id (FK → events, ON DELETE CASCADE), video_path NOT NULL, created_at. Индекс по event_id.

**`workplan`** — планы работы
- id, title NOT NULL, document_path, original_name, **year SMALLINT**, **month SMALLINT**, created_at, updated_at.
- Индекс `idx_workplan_year_month (year DESC NULLS LAST, month DESC NULLS LAST, created_at DESC)`.

**`documents`** — документы
- id, title NOT NULL, document_path, original_name, created_at, updated_at.

**`reminders`** — памятки
- id, title NOT NULL, image_path, created_at, updated_at.

**`clubs`** — клубы и секции
- id, name NOT NULL, leader, created_at, updated_at.

**`schema_migrations`** — служебная: filename PK, applied_at.

### 7.2. Хранение файлов
- Физические файлы хранятся в каталоге `uploads/<ресурс>/` (`events`, `workplan`, `documents`, `reminders`); в БД хранится путь.
- В продакшене каталог смонтирован как именованный том `uploads`; nginx отдаёт его по `/uploads/`.
- При удалении/замене записи соответствующий файл удаляется с диска (с защитой от файлов-сирот при каскадном удалении галереи).
- Тип DATE отдаётся из БД строкой `YYYY-MM-DD` (исключает сдвиг даты из-за таймзоны при сериализации в JSON).

## 8. Программный интерфейс (REST API)

### 8.1. Общие соглашения
- Базовый префикс на проде: `/api` (срезается nginx при проксировании на бэкенд).
- Формат успешного ответа: `{ "data": ... }`; для списков — `{ "data": [...], "total": N }` + заголовок `X-Total-Count`.
- Формат ошибки: `{ "error": { "message": "...", "statusCode": N, "details"?: [...] } }`.
- Аутентификация: JWT в httpOnly cookie `token` (срок 2 часа). **Скользящая сессия**: если до истечения остаётся ≤15 минут, при любом авторизованном запросе выдаётся новый токен.
- Куки: `httpOnly`, `sameSite=strict`, `secure` в продакшене (HTTPS обязателен).
- Пагинация списков: `?page` (по умолч. 1), `?limit` (по умолч. 12, максимум 100).
- Сортировка (где поддерживается): `?sort=<поле>&order=ASC|DESC` через безопасный whitelist полей.

### 8.2. Перечень эндпоинтов

**Служебные**
| Метод | Путь | Доступ | Назначение |
|---|---|---|---|
| GET | `/health` | публичный | Проверка живости (включая доступность БД) |

**Аутентификация** (`/auth`)
| Метод | Путь | Доступ | Назначение |
|---|---|---|---|
| POST | `/auth` | публичный (rate-limit 10/мин) | Вход (login, password) → ставит cookie |
| POST | `/auth/logout` | публичный | Выход (очищает cookie) |
| GET | `/auth/me` | админ | Данные текущего пользователя |

**События** (`/events`)
| Метод | Путь | Доступ | Назначение |
|---|---|---|---|
| GET | `/events` | публичный | Список (пагинация, фильтр `?year`, сортировка) |
| GET | `/events/years` | публичный | Годы со счётчиком для фильтра |
| GET | `/events/:id` | публичный | Событие + галерея + видео |
| POST | `/events` | админ | Создать (multipart; `image` необязателен — без него заглушка) |
| PATCH | `/events/:id` | админ | Изменить |
| DELETE | `/events/:id` | админ | Удалить (с файлами) |
| DELETE | `/events/:id/image` | админ | Сбросить главное фото (`image_path` → NULL) |
| POST | `/events/:id/images` | админ | Добавить изображения (до 10) |

## 9. Нефункциональные требования

### 9.1. Безопасность
1. **Аутентификация** — JWT в httpOnly-cookie (недоступна JS), `sameSite=strict`, `secure` на проде. Пароли — bcrypt (cost 10). Учётка администратора создаётся из docker secret / `ADMIN_PASSWORD_FILE`, в репозиторий не попадает.
2. **Авторизация** — middleware `authenticateToken` + `requireAdmin` на всех изменяющих эндпоинтах.
3. **Ограничение частоты запросов** (rate limiting): вход — 10/мин, обратная связь — 5/мин, общий лимит — 100/мин.
4. **Загрузка файлов — двухуровневая проверка**:
   - multer-фильтр по заявленному MIME и лимиту размера (изображения 5 МБ, документы 20 МБ, видео 200 МБ);
   - проверка **реального** типа по сигнатуре (`file-type`, magic bytes) — защита от подмены Content-Type; при несовпадении файл удаляется, возвращается 400;
   - нормализация расширения сохранённого файла под реальный тип (защита от неверной отдачи Content-Type прокси-сервером).
   - Имена файлов на диске рандомизируются (`timestamp_random.ext`).
   - Разрешённые типы: изображения — JPEG/PNG/WebP; документы — PDF/DOC/DOCX; видео — MP4/WebM/MOV.
5. **Заголовки безопасности** — CSP задаётся в двух местах синхронно (helmet на ответах `/api`, nginx на HTML-страницах; разрешены источники Яндекс.Метрики, 2ГИС, iframe Госуслуг). Дополнительно nginx навешивает `Strict-Transport-Security` (HSTS, `includeSubDomains`), `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy` и скрывает версию сервера (`server_tokens off`).
6. **CORS** — allowlist доменов из `CLIENT_ORIGIN`; запросы без Origin (same-origin, healthcheck) пропускаются.
7. **Прочее** — `helmet` (набор защитных заголовков), `trust proxy = 1` (корректный IP за nginx), параметризованные SQL-запросы (защита от инъекций), безопасная сборка `ORDER BY` по whitelist.
8. **Соответствие 152-ФЗ** — баннер согласия на обработку ПДн, обязательный чекбокс согласия в форме обратной связи.
9. Бэкенд в контейнере работает **не от root** (пользователь `node`, uid 1000).
10. **Postgres не публикуется наружу** — порт проброшен только на `127.0.0.1` (бэкенд ходит в БД по внутренней docker-сети `postgres:5432`); СУБД недоступна из интернета.

### 9.2. Производительность
- Кеширование серверного состояния на клиенте (TanStack Query), длительный кеш редко меняющихся данных (годы событий).
- Оптимизация загрузки: код-сплиттинг админки/логина (lazy chunks), приоритетная загрузка изображений первого ряда (LCP), скелетоны, плавные переходы, `keepPreviousData` при пагинации.
- Пул соединений к БД, индексы под публичные сортировки.
- Отдача статики и файлов напрямую nginx; видео — с поддержкой HTTP Range (206).
- Лимит размера тела запроса на nginx — 210 МБ (под загрузку видео).

### 9.3. Надёжность и отказоустойчивость
- Healthcheck-и контейнеров (postgres `pg_isready`, backend `/health`, зависимости через `depends_on: condition: service_healthy`).
- `restart: unless-stopped` для всех сервисов.
- Корректное завершение (graceful shutdown): закрытие HTTP-сервера и пула БД по SIGTERM/SIGINT с таймаутом-страховкой 10 c.
- Централизованный обработчик ошибок: 5xx логируются, наружу не утекают детали; ошибки multer/валидации → 400.
- Откат файловых операций при ошибках БД (удаление загруженного файла, ROLLBACK транзакции при пакетной загрузке галереи).
- Обработчик ошибок пула БД (`pool.on('error')`): обрыв простаивающего соединения (перезапуск БД, сетевой таймаут) логируется, не роняя процесс.

### 9.4. Доступность (a11y)
- Версия для слабовидящих (BVI, см. 5.1.11).
- Семантическая разметка, skip-link «Перейти к содержимому», перевод фокуса в `<main>` при смене страницы (озвучивание скринридером), `aria-*` атрибуты в формах, `role="alert"`/`role="status"` для сообщений.

### 9.5. Совместимость
- Современные браузеры (desktop и mobile), адаптивная вёрстка (бургер-меню на мобильных).
- Язык интерфейса — русский (`<html lang="ru">`, русская локализация админки).

## 12. Требования к развёртыванию

1. **Платформа**: Linux-сервер (VPS) с установленными Docker и docker compose.
2. **Состав стека** (`docker-compose.yml`): `postgres` (16-alpine), `backend`, `nginx` (собирается из `mrdk-front` — внутри запекается `dist`), `certbot`.
3. **Порты**: nginx — 80 (ACME-проверка + редирект на HTTPS) и 443 (основной).
4. **HTTPS**: сертификат Let's Encrypt через certbot (webroot), автоматическое продление; nginx — TLS 1.2/1.3.
5. **Сборка прод-кода**: бэкенд исполняется из скомпилированного `dist`, фронтенд — из собранного Vite `dist`, запечённого в nginx-образ. Любая правка в `src` попадает в прод только после пересборки образа.
6. **Деплой**: `./deploy.sh` — `git pull --ff-only` → `docker compose build` → `up -d` → ожидание healthcheck бэкенда → очистка висячих образов. Миграции накатываются автоматически при старте бэкенда.
7. **Изменение nginx.conf** не требует пересборки — `nginx.conf` смонтирован bind-mount, достаточно `restart nginx`.
8. **Конфигурация nginx**: отдача SPA (`try_files … /index.html`), отдача `/uploads/`, проксирование `/api/` (с `rewrite` для срезания префикса) и `/health`, CSP-заголовок, `client_max_body_size 210M`.

## 13. Эксплуатация, резервное копирование и мониторинг

- **Резервное копирование** (`backup.sh`): дамп БД (`pg_dump | gzip`) + архив тома `uploads` (tar.gz), ротация по сроку хранения (по умолчанию 14 дней), пригоден для cron. Каталог бэкапов защищён правами `700`.
- **Восстановление** (`restore.sh`): восстановление БД и загрузок из бэкапа.
- **Логирование**: winston (JSON в проде) + morgan (HTTP-запросы); вывод в stdout контейнера.
- **Мониторинг живости**: эндпоинт `/health` и docker healthcheck-и.
- Подробный рунбук (HTTPS-сертификат, грабли nginx-прокси, права на том загрузок, синхронизация CSP) — [OPERATIONS.md](OPERATIONS.md).

## 15. Известные ограничения и направления развития

1. **Один уровень доступа** — только администратор; разграничение прав (редактор/модератор) не реализовано.
2. **Нет механизма сидирования** данных — наполнение через админку или разовый импорт.
3. **Двойная CSP** (helmet + nginx) требует ручной синхронизации при добавлении внешних источников.
4. **Вход в админку требует HTTPS** — cookie с флагом `secure` не сохраняется по plain HTTP (кроме `http://localhost`); на проде это закрывается TLS.
5. Возможные направления развития: полнотекстовый поиск по событиям, расширенная ролевая модель, экспорт обращений/логирование обратной связи в БД, CDN для медиа.
